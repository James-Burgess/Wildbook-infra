Kedro Best Practices

This page covers Kedro project setup, directory structure, and naming conventions based on the official Kedro documentation. Following these best practices ensures that your pipeline remains maintainable, reproducible, and easy for other team members to understand.

Project Setup

The recommended way to bootstrap a new Kedro project is using uvx kedro new, which runs Kedro in an isolated environment:

uvx kedro new --starter spaceflights-pandas --name spaceflights
cd spaceflights
uv sync
uv run kedro run --pipeline __default__

Alternative methods include pipx run kedro new, a global pip install, or conda:

# Using pipx
pipx run kedro new

# Using conda
conda create --name kedro-environment python=3.13 kedro --yes
conda activate kedro-environment
kedro new

Kedro supports Python 3.10+. The core Kedro Framework supports all Python versions actively maintained by CPython. The kedro-datasets package follows NEP 29, meaning it may drop Python versions faster than the core framework.

Standard Project Structure

A Kedro project with all tools selected has this layout:

project-dir/
├── conf/             # Configuration (base + local overrides)
│   ├── base/         # Shared project settings
│   │   ├── catalog.yml
│   │   ├── parameters.yml
│   │   └── logging.yml        # When Log tool is selected
│   └── local/        # User-specific, git-ignored
│       └── credentials.yml
├── data/             # Project data (not committed to version control)
│   ├── 01_raw/
│   ├── 02_intermediate/
│   ├── 03_primary/
│   └── 04_report/
├── docs/             # Sphinx/Sphinx-based documentation
├── notebooks/        # Jupyter notebooks for experiments
├── src/
│   └── <package_name>/
│       ├── __init__.py
│       ├── __main__.py
│       ├── settings.py
│       ├── pipeline_registry.py
│       └── pipelines/       # One folder per pipeline
├── tests/            # Unit and integration tests
├── .gitignore
├── pyproject.toml    # Project metadata and build config
├── README.md
└── requirements.txt

When no optional tools are selected, the data/, docs/, and tests/ directories are omitted.

Tool Selection

During kedro new, you can choose which tools to include:

  • Lint (Ruff) — Adds [tool.ruff] configuration to pyproject.toml
  • Test (Pytest) — Adds a tests/ directory with testing infrastructure
  • Log — Adds logging.yml inside conf/ for customised logging
  • Docs (Sphinx) — Adds a docs/ folder with Sphinx setup
  • Data Folder — Creates a data/ directory with sub-folders for different pipeline stages
  • PySpark — Adds PySpark-specific configuration files

Naming Conventions

Nodes

  • Node names must only contain letters, digits, hyphens, underscores, and periods
  • Use descriptive names that convey the business logic: name="adding_a_and_b" rather than name="node1"
  • Avoid using dot (.) notation in dataset names — dots indicate namespace pipelines internally. Use underscores instead: input1_kedro not input1.kedro

Pipelines

  • Organise pipelines using the principle: one pipeline, one folder
  • Each pipeline module must expose a create_pipeline(**kwargs) -> Pipeline function
  • Use tags to group nodes: tags="data_processing"
  • The __default__ pipeline is automatically the sum of all registered pipelines

Tests

  • Mirror the src/<package_name> directory structure under tests/
  • Name test files test_<file_being_tested>.py
  • Name test functions test_<descriptive_name> and classes Test<Name>

Configuration

  • Shareable config goes in conf/base/
  • Secrets and personal settings go in conf/local/ (git-ignored)
  • Keep pipeline_registry.py and settings.py in src/<package_name>/

Configuration Best Practices to Avoid Leaking Data

  • Do not commit data to version control — add data/ to .gitignore
  • Do not commit notebook output cells
  • Keep sensitive credentials in conf/local/credentials.yml
  • Use conf/local/catalog.yml to override production catalog entries with local paths during development

Data Layering Convention

Kedro encourages a layered approach to data processing. By convention, data files are organised as:

LayerDescription
01_rawOriginal, immutable data dumps
02_intermediatePreprocessed and cleaned data
03_primaryFeature-engineered or transformed data
04_reportOutputs, models, reports

In Kedro-Viz, this layering is further visualised by adding a layer attribute under metadata.kedro-viz in catalog.yml.

Key Files

pyproject.toml
Defines project metadata, dependencies, and the [tool.kedro] section with package_name, source_dir, and kedro_init_version. Also stores Ruff and pytest configuration.
settings.py
Controls CONF_SOURCE, CONFIG_LOADER_ARGS, hooks registration, session class, and catalog class.
pipeline_registry.py
Must expose a register_pipelines() function that returns dict[str, Pipeline].

Core Concepts

Node
A wrapper for a pure Python function that names its inputs and outputs. Nodes are the building blocks of pipelines.
Pipeline
Organises the dependencies and execution order of a collection of nodes. The pipeline resolves data dependencies automatically — nodes do not necessarily run in declaration order.
Data Catalog
A registry of all data sources. Maps node input/output names to dataset classes with their storage configuration.
KedroSession
Manages the lifecycle of a Kedro run, persisting runtime parameters (CLI flags, environment variables).
KedroContext
Holds configuration and Kedro's main functionality.
