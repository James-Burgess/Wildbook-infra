What's Missing

Based on Kedro starter conventions, configuration best practices, deployment guidelines, and production Python data project standards, here is what a well-structured Kedro project would include that the current wildbook_pipeline does not yet have.

1. Project Setup & Scaffolding

A Kedro project created from an official starter (e.g. spaceflights-pandas) includes a richer baseline than kedro new alone. Starters are Cookiecutter templates that bundle tests, CI configs, and environment-aware configuration from day one.

MISSING .dockerignore — Prevents sending venv/, __pycache__/, data/, and IDE files into the Docker build context. Without it, every COPY . . in the Dockerfiles includes hundreds of megabytes of unnecessary files, making builds slow and images large.
Suggested .dockerignore 
venv/
.git/
.viz/
data/
logs/
__pycache__/
*.pyc
*.egg-info/
.ipynb_checkpoints/
.DS_Store
.idea/
.vscode/
*.log
MISSING .pre-commit-config.yaml — Pre-commit hooks for auto-formatting, linting, and import sorting on every commit. Standard in Kedro starter templates.
Suggested .pre-commit-config.yaml 
repos:
  - repo: https://github.com/psf/black
    rev: 24.4.0
    hooks:
      - id: black
        language_version: python3.12
  - repo: https://github.com/pycqa/isort
    rev: 5.13.2
    hooks:
      - id: isort
        args: [--profile, black]
  - repo: https://github.com/pycqa/flake8
    rev: 7.0.0
    hooks:
      - id: flake8
MISSING setup.cfg (or [tool.flake8] in pyproject.toml) — Flake8 and isort configuration for consistent linting. The Makefile lint target runs flake8 src/ but there is no configuration file controlling which rules are enforced or ignored.
MISSING LICENSE — Standard for any open-source or shared project. Missing from the repo entirely.
MISSING py.typed marker file — Placed at src/wildbook_pipeline/py.typed to declare that the package supports PEP 561 type hints, enabling downstream type checkers.
INCOMPLETE requirements.txt — Only 6 packages listed with loose version specifiers. A production project should:
  • Pin exact versions with hashes in requirements.txt
  • Maintain separate requirements-dev.txt for dev tools (black, flake8, mypy, pre-commit)
  • Use pip-compile (pip-tools) from a requirements.in for reproducible builds
  • Dockerfile references kedro-datasets[pandas] but pyproject.toml does not list it in [project] dependencies
INCOMPLETE CHANGELOG.md — Important for tracking breaking changes in a production data project.

2. Configuration

Kedro uses the OmegaConfigLoader (OmegaConf-based) to load config from conf/ following specific naming patterns. The loader recursively scans conf/base first, then conf/local (the default overriding environment), merging the results. Configuration files are matched by patterns: "catalog", "parameters", "credentials", and "logging".

MISSING / BUG conf/base/logging.yml — Every Kedro project should have a logging configuration file. More importantly, the Dockerfile references this file: 
ENV KEDRO_LOGGING_CONFIG=conf/base/logging.yml
The file does not exist in the repository, which will cause errors at runtime. Both Dockerfile and Dockerfile.airflow set this environment variable.
Minimal logging.yml (for conf/base/) 
version: 1
disable_existing_loggers: false
formatters:
  simple:
    format: "%(asctime)s - %(name)s - %(levelname)s - %(message)s"
handlers:
  console:
    class: logging.StreamHandler
    level: INFO
    formatter: simple
    stream: ext://sys.stdout
loggers:
  kedro:
    level: INFO
    handlers: [console]
    propagate: false
root:
  level: INFO
  handlers: [console]
MISSING Environment-specific configuration (conf/prod/, conf/test/) — Kedro supports arbitrary configuration environments. A production project typically has:
  • conf/prod/parameters.yml — production-specific overrides (batch sizes, memory limits, cloud paths)
  • conf/test/parameters.yml — test-specific overrides (smaller datasets, mock endpoints)
  • conf/dev/parameters.yml — development-specific config
Currently only base (foundation) and local (gitignored overrides) exist. You can use these with kedro run --env=prod or set KEDRO_ENV=prod.
MINIMAL conf/base/parameters.yml — Contains a single parameter (images_path). A real pipeline should have: model hyperparameters, batch sizes, feature extraction parameters, output paths, and runtime flags. Consider splitting into parameters_data.yml, parameters_model.yml, etc. as the config patterns support parameters* wildcards.
MISSING conf/base/credentials.yml — Kedro's config loader looks for credentials files matching credentials* patterns. Having at least a stub in base (with no real secrets) documents the expected credential structure. Real secrets go in local (gitignored).
MISSING Remote configuration support — Kedro supports loading config from S3, GCS, Azure Blob. For a production Airflow deployment, consider storing conf/prod/ on cloud storage so that config can be updated without redeploying containers. 
kedro run --conf-source=s3://my-bucket/configs/

3. Testing

MISSING tests/ directory — The project has zero tests. Every Kedro starter includes a tests/ directory with at least:
  • tests/test_pipelines.py — pipeline registration tests
  • tests/test_nodes.py — unit tests for individual node functions
  • tests/conftest.py — shared fixtures (catalog, config, dummy data)
  • tests/test_config.py — config loading validation
The Makefile test target runs kedro test which expects a tests/ directory with pytest fixtures.
Minimal test structure 
tests/
├── conftest.py              # Pytest fixtures (dummy catalog, project context)
├── test_pipelines.py        # Pipeline structure tests
└── pipelines/
    └── example/
        └── test_nodes.py    # Unit tests for each node function
MISSING CI/CD pipeline configuration — No .github/workflows/, no .gitlab-ci.yml, no Jenkinsfile. A production project should have automated test runs on every PR, with linting, type checking, and integration tests.
MISSING Code coverage tooling — The .gitignore already ignores .coverage, htmlcov/, and coverage.xml, but there is no pytest-cov or coverage dependency, nor any coverage configuration. Add pytest-cov to dev requirements and [tool.coverage] to pyproject.toml.

4. Packaging & Deployment

Kedro recommends container-based deployment (which this project has), but also supports package-based deployment via kedro package for distributing .whl files. The official docs also suggest the Kedro-Docker plugin for streamlined Docker workflows.

MISSING kedro-docker plugin — The official Kedro docs recommend using the Kedro-Docker plugin for streamlined Docker workflows. It auto-generates optimized Dockerfiles, manages entrypoints, and integrates with kedro package. 
# Add to requirements.txt:
kedro-docker>=0.5.0
MISSING kedro package workflow — The kedro package command builds a .whl file containing pipeline source code and a tar.gz with configuration (excluding conf/local/). The current Dockerfile uses pip install -e . which is a development install, not a proper packaged distribution. The packaging docs recommend kedro package for production distributions.
MISSING Multi-stage Dockerfile — The production Dockerfile does a single-stage build with pip install -e . and includes all source and build tools in the final image. A multi-stage build would produce a smaller, more secure image by separating build dependencies from runtime dependencies.
MISSING Health checks for the Kedro container — The docker-compose.prod.yml includes health checks for Airflow, Postgres, and Redis services but not for the Kedro pipeline container itself.
INCOMPLETE Makefile clean target — The clean target removes data/ which could accidentally delete valuable data. A safer approach is to only clean data/02_intermediate/ and data/03_primary/ (Kedro's convention for transient outputs).

5. Code Quality & Type Safety

MISSING mypy / type checker configuration — The pipeline only uses from __future__ import annotations in a few files. There is no [tool.mypy] section in pyproject.toml and no mypy in requirements.
Suggested [tool.mypy] config 
[tool.mypy]
python_version = "3.10"
strict = false
warn_return_any = true
warn_unused_configs = true
ignore_missing_imports = true
exclude = ["venv/", "build/"]
MISSING Black formatter config in pyproject.toml — The project uses Flake8 and isort for linting but has no auto-formatter configured. Add [tool.black] to pyproject.toml: 
[tool.black]
line-length = 88
target-version = ["py310"]
MISSING isort config in pyproject.toml — The Makefile runs isort --profile black src/ but there's no [tool.isort] in pyproject.toml to make the config explicit and IDE-friendly. 
[tool.isort]
profile = "black"
line_length = 88

6. Observability & Monitoring

MISSING Structured logging — The missing conf/base/logging.yml means there is no logging configuration. A production pipeline should log run metadata: pipeline name, node execution times, dataset sizes, and error details. Consider adding JSON-format logging for ingestion into log aggregators (e.g. ELK, Datadog).
MISSING Pipeline metrics / hooks — Kedro supports project hooks for instrumentation. There is no hooks.py file in src/wildbook_pipeline/. Consider adding hooks for:
  • Node execution timing (performance monitoring)
  • Dataset I/O statistics (rows processed, file sizes)
  • Pipeline run status notifications (Slack, email)
  • Error tracking integration (Sentry)
MISSING Run tracking / experiment lineage — No integration with MLflow, Weights & Biases, or Kedro's built-in run tracking for experiment tracking and artifact lineage across pipeline runs.

7. Documentation

INCOMPLETE Pipeline documentation — The existing docs/ directory has good static HTML pages covering the architecture, but there is no:
  • API documentation generated from docstrings (Sphinx / MkDocs with mkdocstrings)
  • Data schema documentation for the CSVs flowing between nodes
  • Kedro-Viz embedded pipeline diagram
  • Troubleshooting guide / FAQ
MISSING docstrings in node functions — The src/wildbook_pipeline/pipelines/example/nodes.py file has four functions with zero docstrings. Every node function should document its inputs, outputs, transformation logic, and any side effects.

8. Security

MISSING Secrets management — The .env.example is minimal and docker-compose.prod.yml hardcodes passwords (POSTGRES_PASSWORD: airflow, hardcoded Airflow webserver secret key). A production setup should use Docker secrets, Kubernetes secrets, or a secrets manager (AWS Secrets Manager, HashiCorp Vault).
MISSING Dependency vulnerability scanning — No pip-audit, safety, or Dependabot configuration. Dependencies should be scanned regularly for known CVEs, especially for a production Airflow deployment.

9. Data Management

MISSING Data versioning — No integration with DVC, Delta Lake, or Kedro's dataset versioning. Raw data in data/01_raw/ has no version tracking, making it difficult to reproduce results across pipeline runs.
MISSING Data validation / Great Expectations — No schema validation for input/output datasets. Kedro integrates with data quality tools for validating pipeline inputs and outputs (checking column types, null rates, value ranges, etc.).
INCOMPLETE Catalog dataset types — All four datasets in conf/base/catalog.yml use the same pandas.CSVDataset type with local filepaths. For production (especially with Airflow/Celery distributed workers), consider:
  • Cloud storage backends (pandas.S3Dataset, pandas.GCSDataset)
  • Partitioned datasets for large data (kedro.io.PartitionedDataset)
  • Incremental dataset loading for checkpoint/resume workflows
  • Adding layer tags (raw, intermediate, primary, reporting) for catalog organization

10. Airflow Deployment Gaps

MISSING Kedro-Airflow integration plugin — The dags/wildbook_pipeline.py manually wraps Kedro in a BashOperator. Kedro offers an official kedro-airflow plugin that auto-generates Airflow DAGs from your pipeline nodes, supporting proper error handling and retries at the node level instead of the entire pipeline level.
MISSING Airflow Variables/Connections config — The Airflow DAG should use Airflow Variables and Connections for configuration instead of relying entirely on conf/ yaml files. This enables dynamic configuration changes from the Airflow UI.
MISSING Airflow Fernet key — AIRFLOW__CORE__FERNET_KEY: "" is set to empty in docker-compose.prod.yml. This should be a generated key for encrypting connection secrets.
