Kedro Deployment Patterns

This page covers Kedro's deployment patterns and how they apply to the Wildbook Pipeline. Kedro officially supports both single-machine and distributed deployment strategies.

Deployment Decision Tree

Kedro provides a decision framework for choosing the right deployment approach:

  1. Can your pipeline run on a single machine?
    • Yes & Docker available: Use container-based deployment
    • Yes & No Docker: Use CLI or package-based deployment
  2. Need to run across separate machines?
    • Consult the distributed deployment guides for your platform of choice

Single-Machine Deployment

Container-Based (Docker)

The recommended approach for reproducible single-machine deployment. Our Wildbook Pipeline uses this pattern:

# Dockerfile
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
RUN pip install --no-cache-dir -e .
ENV KEDRO_LOGGING_CONFIG=conf/base/logging.yml
CMD ["kedro", "run"]

Build and run:

docker build -t wildbook-pipeline .
docker run wildbook-pipeline

# Or with docker-compose:
docker compose up -d

Key considerations for Docker deployment:

  • Use ENV to configure KEDRO_LOGGING_CONFIG for container-aware logging
  • Mount data directories as volumes for persistent storage
  • Set KEDRO_CONFIG_LOADER_ARGS to select the correct run environment
  • Leverage multi-stage builds to keep images small

Package-Based Deployment

If Docker is not available, deploy the Kedro project as a standard Python package:

# Build
pip install build
python -m build

# Deploy
pip install dist/wildbook_pipeline-*.whl
# Run
kedro run --pipeline example

CLI-Based Deployment

Clone and run directly on the target server:

git clone <repository-url>
cd pipeline
python -m venv venv
source venv/bin/activate
pip install -e .
kedro run --pipeline example

Distributed Deployment

Kedro supports integration with several distributed platforms. The Wildbook Pipeline uses Apache Airflow for production scheduling:

Apache Airflow

Our production stack uses Airflow with CeleryExecutor. Key files:

# Dockerfile.airflow
FROM apache/airflow:2.10.5-python3.12
# ... installs Kedro and project dependencies ...

# dags/wildbook_pipeline.py
from airflow import DAG
from airflow.operators.bash import BashOperator
from datetime import datetime

with DAG(
    dag_id="wildbook_pipeline",
    schedule="@daily",
    start_date=datetime(2024, 1, 1),
    catchup=False,
) as dag:
    run_kedro = BashOperator(
        task_id="run_kedro_pipeline",
        bash_command="cd /opt/airflow/wildbook-pipeline && kedro run --pipeline example",
    )

The Airflow stack consists of:

  • PostgreSQL — Metadata database
  • Redis — Celery broker and result backend
  • Airflow Webserver — UI on port 8080
  • Airflow Scheduler — DAG scheduling
  • Airflow Worker — Celery task execution
  • Airflow Triggerer — Deferrable operator support

Hooks in Deployment

Hooks allow injecting custom behaviour at specific lifecycle points, which is especially useful in production:

  • before_pipeline_run / after_pipeline_run — Add logging, metrics emission, or slack notifications
  • after_catalog_created — Validate catalog configuration
  • on_node_error / on_pipeline_error — Alert on failures, send to monitoring systems
  • before_dataset_saved / after_dataset_saved — Data quality checks, lineage tracking

Register hooks in src/<package_name>/settings.py:

# src/wildbook_pipeline/hooks.py
from kedro.framework.hooks import hook_impl

class MonitoringHooks:
    @hook_impl
    def after_pipeline_run(self, run_params, pipeline, catalog):
        # Send metrics to monitoring system
        send_metrics(pipeline_name=str(pipeline))

    @hook_impl
    def on_node_error(self, error, node):
        # Alert on node failure
        send_alert(f"Node {node.name} failed: {error}")

# src/wildbook_pipeline/settings.py
from wildbook_pipeline.hooks import MonitoringHooks
HOOKS = (MonitoringHooks(),)

Hooks follow LIFO (last-in, first-out) execution order. Auto-discovered hooks from installed plugins run first, followed by project-defined hooks.

Logging in Deployment

Kedro uses Python's logging library with Rich formatting by default. Customise for deployment via KEDRO_LOGGING_CONFIG:

export KEDRO_LOGGING_CONFIG=conf/base/logging.yml

Common deployment logging patterns:

  • File-based logging: Add info_file_handler to root handlers (rotating 10MB files)
  • Plain console: Swap rich handler for console in CI/CD environments
  • DEBUG level: Change your_python_package.level from INFO to DEBUG for troubleshooting
  • Rich in CI: Set export COLUMNS=120 LINES=25 for proper formatting in CI logs

Node Grouping for Deployment

Effective grouping strategies improve deployment manageability:

  • By pipeline: Organise related nodes into separate modular pipelines
  • By tags: Tag nodes to enable selective execution (kedro run --tags=processing)
  • By namespaces: Use namespace pipelines (dot.notation) to scope datasets and avoid collisions

Environment Management

Use conf/base/ for production configuration and override in conf/local/ for development. Select the runtime environment via settings:

# settings.py
CONFIG_LOADER_ARGS = {
    "base_env": "base",
    "default_run_env": "local",
}

Kedro-Viz in Deployment

Kedro-Viz provides pipeline visualisation and can be run as a standalone service:

# Docker-based
docker compose up -d viz    # runs on port 4141

# Direct
kedro viz run               # runs on http://localhost:4141

It helps understand pipeline structure, search nodes and datasets, and visualise sub-pipelines via tags.
