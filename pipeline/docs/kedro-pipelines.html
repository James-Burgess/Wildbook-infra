Kedro Pipelines: Nodes and Patterns

Nodes are the building blocks of Kedro pipelines. A node is a wrapper for a Python function that names its inputs and outputs. A pipeline organises the dependencies and execution order of a collection of nodes.

Creating Nodes

A node is created with Node(func, inputs, outputs). It requires at least one input or output:

from kedro.pipeline import Node

def add(x, y):
    return x + y

adder_node = Node(func=add, inputs=["a", "b"], outputs="sum")
adder_node = Node(func=add, inputs=["a", "b"], outputs="sum", name="adding_a_and_b")

The name argument is optional but recommended — it appears in logs and Kedro-Viz:

print(str(adder_node))
# adding_a_and_b: add([a,b]) -> [sum]

Running Nodes

Nodes can be run individually by passing a dictionary of input values:

adder_node.run(dict(a=2, b=3))
# {'sum': 5}

# Equivalent shorthand
adder_node(dict(a=2, b=3))

Advanced Node Patterns

*args Nodes

For functions that take a variable number of inputs:

def combine_datasets(*args):
    # args receives data for all named inputs
    return do_something(args)

Node(func=combine_datasets, inputs=["df1", "df2", "df3"], outputs="combined")

**kwargs Nodes

When you need to know dataset names at runtime (common for reporting):

def reporting(**kwargs):
    result = []
    for name, data in kwargs.items():
        res = example_report(name, data)
        result.append(res)
    return combined_report(result)

uk_reporting_node = Node(
    reporting,
    inputs={"uk_input1": "uk_input1", "uk_input2": "uk_input2"},
    outputs="uk",
)

A helper pattern to avoid repeating mappings:

mapping = lambda x: {k: k for k in x}

uk_reporting_node = Node(
    reporting,
    inputs=mapping(["uk_input1", "uk_input2"]),
    outputs="uk",
)

Generator Nodes

For large datasets, use Python generators to lazily load/save data. Update catalog.yml with load_args.chunksize and implement a custom dataset with an append-mode _save method:

def make_predictions(X_train, X_test, y_train) -> Generator:
    model = DecisionTreeClassifier()
    model.fit(X_train, y_train)
    for chunk in X_test:
        y_pred = model.predict(chunk)
        yield pd.DataFrame(y_pred)

Preview Functions (Experimental)

Preview functions inject debugging/monitoring callbacks. Supported types: MermaidPreview (diagrams), ImagePreview (plots), TextPreview (summaries), and CustomPreview:

from kedro.pipeline.preview_contract import MermaidPreview

def preview_training_model() -> MermaidPreview:
    return MermaidPreview(
        content="""
        flowchart TD
            A[Load Dataset] --> B[Train Model]
            B --> C[10 Epochs]
            C --> D[Status: Completed]
        """,
        meta={"framework": "sklearn"}
    )

node(func=train_model, inputs="data", outputs="metrics",
     preview_fn=preview_training_model, name="train_node")

Building Pipelines

Pipelines resolve dependencies automatically — nodes are not necessarily run in declaration order:

from kedro.pipeline import Pipeline, node

pipeline = pipeline([
    node(func=load_raw_images, inputs="params:images_path", outputs="raw_images", name="load_node"),
    node(func=preprocess_images, inputs="raw_images", outputs="preprocessed_images", name="preprocess_node"),
    node(func=extract_features, inputs="preprocessed_images", outputs="features", name="extract_node"),
    node(func=generate_report, inputs="features", outputs="report", name="report_node"),
], tags="example")

Pipeline Utilities

  • pipeline.describe() — Print execution order summary
  • pipeline.nodes — Access nodes in topological order
  • pipeline.inputs() / pipeline.outputs() — Get pipeline-level I/O
  • pipeline.from_inputs() — Slice from specific inputs
  • pipeline.to_nodes() — Slice to specific output nodes
  • pipeline.only_nodes_with_tags() — Filter by tags

Tagging

Tags enable running subsets of pipelines without code changes:

# Tag a single node
Node(func=add, inputs=["a", "b"], outputs="sum", tags="node_tag")

# Tag an entire pipeline (applies to all nodes within)
pipeline = Pipeline([...], tags="pipeline_tag")

# Run using tags
kedro run --tags=pipeline_tag
kedro run --tags=node_tag

Merging Pipelines

Multiple pipelines can be combined; their underlying nodes are expanded and merged:

pipeline_all = Pipeline([pipeline_de, pipeline_ds, last_node])
print(pipeline_all.describe())

Avoiding Bad Pipelines

  • No-input-no-output nodes: Invalid — every node must have at least one input or output.
  • Circular dependencies: Kedro detects and rejects cyclic graphs.
  • Dot notation in names: Avoid . in dataset/input names — it indicates a namespace pipeline internally. Use _ instead.

Pipeline Registry

Each pipeline module under pipelines/ must expose create_pipeline(**kwargs) -> Pipeline. The registry auto-discovers them:

# src/wildbook_pipeline/pipeline_registry.py
from kedro.framework.project import find_pipelines
from kedro.pipeline import Pipeline

def register_pipelines() -> dict[str, Pipeline]:
    pipelines = find_pipelines(raise_errors=True)
    pipelines["__default__"] = sum(pipelines.values())
    return pipelines

Modular Pipelines

For larger projects, use modular pipelines — logically isolated, reusable pipeline units stored in separate folders. They are easier to develop, test, and maintain, and can be copied between projects.

Runners

Kedro provides three built-in runners:

  • SequentialRunner — Executes nodes one-by-one in dependency order
  • ParallelRunner — Uses multiprocessing for concurrent execution
  • ThreadRunner — Uses threading for concurrent execution

Project Code Organization

Always create a module under src/<package_name>/pipelines/ with a create_pipeline() function. Each pipeline gets its own folder:

src/wildbook_pipeline/pipelines/
├── __init__.py
├── data_processing/

│   └── pipeline.py
├── data_science/

│   ├── nodes.py
│   └── pipeline.py
└── example/
    ├── __init__.py
